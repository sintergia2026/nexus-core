# NEXUS™ Core

Canonical sovereign core for the NEXUS™ domain-driven operational intelligence system.

---

## 1. Purpose

`nexus-core` is the canonical architectural and operational nucleus of NEXUS™.

This repository is not intended to behave as a dashboard-first application or as a UI-centered analytics product.  
Its purpose is to establish the sovereign system core where operational truth is defined, interpreted, and projected.

The truth of the system lives in:

- registries
- contracts
- ontology
- twin
- engine
- snapshots
- reporting outputs

The interface is not the system.  
The twin is not decoration.  
The report is not the source of truth.  
NEXUS™ is built as a domain-driven operational intelligence system.

---

## 2. Phase 1 Scope

Phase 1 is intentionally narrow.

Its objective is to prove that NEXUS™ can process a weekly operational payload and transform it into a valid operational interpretation.

### Phase 1 pipeline

Operational Intake Payload  
→ Normalization  
→ Metric Calculation  
→ Diagnostic Interpretation  
→ Twin State Assembly  
→ Diagnostic Snapshot  
→ Weekly Report

### Phase 1 output goal

Phase 1 must be able to generate, at minimum:

- a normalized operational payload
- 6 required operational metrics
- diagnostic signals
- a twin state
- a diagnostic snapshot
- a weekly report
- persistent JSON outputs in the examples directory
- a passing integration test

---

## 3. What Phase 1 Currently Does

The current Phase 1 core is capable of:

1. receiving a structured weekly operational payload
2. validating and normalizing the payload
3. computing required Phase 1 metrics
4. interpreting operational signals
5. deriving constraints
6. assigning a twin state
7. generating a diagnostic snapshot
8. generating a weekly report
9. saving JSON artifacts to disk
10. passing a minimum integration test
11. distinguishing between stable and constrained operational scenarios

- constraint_signal

These signals provide the interpretive layer between raw metrics and system state.

---

## 6. Core Phase 1 Outputs

### Twin State

The twin represents the current operational state of a site for a given operational week.

It includes:

- canonical identity
- metrics
- signals
- constraints
- diagnostic summary
- decision output

### Diagnostic Snapshot

The diagnostic snapshot is the frozen evidence object generated from the diagnostic cycle.

It includes:

- twin reference
- metrics
- signals
- constraints
- diagnostic summary
- decision output
- source payload IDs
- version

### Weekly Report

The weekly report is the institutional projection of the snapshot.

It includes:

- executive summary
- metric overview
- signal overview
- constraints overview
- priority actions

The report is a projection of the twin state.  
It is not the source of truth.

## 8. Current Validated Example Scenarios

Phase 1 has currently been validated against at least two example scenarios:

### Constrained scenario

A restaurant case with:

- elevated latency
- material revenue leakage
- moderate volatility
- reporting weakness
- structural coordination pressure

Expected result:

- constrained state
- corrective focus
- priority P2

### Stable scenario

A restaurant case with:

- low latency
- low leakage
- low volatility
- healthy reporting reliability
- no major active constraints

Expected result:

- stable state
- monitor and preserve
- priority P3

This contrast validation confirms that the diagnostic core is capable of distinguishing operational health states.


## 15. Recommended Next Step After Phase 1

The next disciplined step after this Phase 1 closure is **not** UI expansion.

The recommended next step is one of the following:

1. strengthen output contracts and artifact stability
2. introduce additional contrast scenarios
3. define canonical output bundles for operational delivery
4. prepare a structured Phase 2 design document
5. only then evaluate persistence, API exposure, or controlled sector expansion

The correct sequence remains:

- contracts first
- ontology first
- twin before interface
- metrics before narrative
- governance before automation
